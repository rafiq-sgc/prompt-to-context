Why This Chapter Exists

Chapter 2 gives a production-focused recap — what seniors must know to control AI. This chapter is the complete reference: how models are built, how they run, what knobs you can turn, and how to get reliably better output.

Read this chapter if you want to understand…

  • What happens inside the model when you send a prompt
  • Why the same prompt gives different answers on different runs
  • Which model and settings to pick for SQL vs brainstorming vs code review
  • Why context engineering matters more as models get larger
An LLM does not "think." It predicts the next token. Everything else — usefulness, danger, productivity — comes from how you frame the task, supply context, and verify output.

Part 2 — Tokenization: The Hidden Interface

Before any "intelligence," your text becomes a sequence of tokens. Tokenization is model-specific — GPT-4, Claude, and Llama do not split text identically.

# Same English, different token counts (illustrative):
"Hello world"           → 2 tokens (common phrase)
"student_enrollment"    → 2–4 tokens depending on model
"反"                    → 1 token in multilingual models
"SELECT"                → often 1 token (frequent in training data)
"Supercalifragilistic"  → many sub-word tokens

Common Tokenization Methods

MethodHow it worksExample
BPE (Byte Pair Encoding)Merge frequent character pairs iterativelyenrollment → enroll + ment
SentencePieceLanguage-agnostic subword unitsUsed by Llama, many open models
Tiktoken-styleOptimized BPE for code + multilingualOpenAI GPT family

Why Tokenization Matters to Developers

  • Cost: APIs charge per input + output token — long prompts are expensive
  • Context limits: A "200K context" window is 200K tokens, not words (~75K words roughly)
  • Code: CamelCase and snake_case split differently — affects few-shot examples
  • JSON: Whitespace and braces consume tokens — compact schemas help
  • Unicode bugs: Invisible characters can change tokenization and behavior
🎤 Demo: Paste the same 500-line file into a tokenizer tool (or ask the model "how many tokens is this?"). Show how schema dumps blow the budget — bridges to RAG (Ch 7).

Part 3 — Embeddings: Words as Numbers

Each token ID maps to an embedding vector — typically 1,024 to 16,384 dimensions. Similar meanings cluster nearby in this space (approximately).

flowchart LR T1["Token: student"] --> V1["Vector: [0.2, -0.8, ...]"] T2["Token: pupil"] --> V2["Vector: [0.19, -0.79, ...]"] T3["Token: banana"] --> V3["Vector: [-0.5, 0.3, ...]"] V1 -.similar.-> V2 V1 -.far.-> V3

Generative LLMs use embeddings internally during generation. Embedding models (OpenAI text-embedding-3, Cohere, open-source encoders) output vectors you store for RAG search — they do not generate text.

Model typeInputOutputUse case
Generative LLMText promptMore textChat, code, SQL drafts
Embedding modelText chunkFixed-size vectorSemantic search, RAG index
RerankerQuery + candidate docsRelevance scoreImprove RAG precision

Part 4 — Transformer Architecture (What Seniors Should Know)

Modern LLMs are stacks of transformer decoder blocks. You do not need to implement them — but you need the mental model.

flowchart TB subgraph One Transformer Block IN[Token embeddings + position info] ATT[Multi-head self-attention] ADD1[Residual + layer norm] FFN[Feed-forward network] ADD2[Residual + layer norm] OUT[Output to next layer] end IN --> ATT --> ADD1 --> FFN --> ADD2 --> OUT

Self-Attention — "Which tokens matter for this token?"

Every token can attend to every other token in the context window. When generating SQL, the model weighs relationships between SELECT, table names in your schema chunk, and the user question — simultaneously.

  • Query, Key, Value: Internal mechanism for computing relevance weights between token pairs
  • Multi-head: Multiple attention patterns in parallel (syntax, semantics, long-range links)
  • Causal mask: During generation, future tokens are hidden — model only sees past tokens

Scale: Parameters and Layers

TermMeaningPractical note
ParametersLearned weights (billions)Larger ≠ always better for your task; fit and latency matter
LayersDepth of transformer stackMore depth = more abstraction capacity
Hidden dimensionWidth of internal representationsAffects memory during inference
Context lengthMax tokens in one requestLong context costs more compute (attention is O(n²) naively)

Part 5 — How Models Are Trained (Three Phases)

flowchart LR P1[Phase 1: Pretraining] P2[Phase 2: Supervised Fine-Tuning] P3[Phase 3: Alignment RLHF/DPO] P1 --> P2 --> P3 --> DEPLOY[Deployed Chat Model]

Phase 1 — Pretraining

Model reads massive text corpora (web, books, code, papers) and learns to predict the next token. It absorbs grammar, facts (with cutoff date), coding patterns, and reasoning style — but no instruction-following yet.

Phase 2 — Supervised Fine-Tuning (SFT)

Human-curated examples: (instruction, ideal response) pairs. Teaches the model to act as a helpful assistant, follow formats, refuse some requests, write code in blocks.

Phase 3 — Alignment (RLHF / DPO / similar)

Human raters rank outputs; model is adjusted to prefer helpful, harmless, honest responses. This is why chat models behave differently from raw base models.

PhaseWhat the model learnsWhat it does NOT fix
PretrainingLanguage, world patterns, code syntaxYour private schema, today's API docs
SFTInstruction following, chat formatGuaranteed factual accuracy
AlignmentHelpful tone, safety refusalsHallucination elimination
Training teaches plausible text. Your job as a developer is to supply true context and verify correct behavior.

Part 6 — Inference: How Text Is Generated at Runtime

Autoregressive Decoding

The model generates one token at a time. Each new token is appended to the input; the full sequence is re-processed (with optimizations). This is why:

  • Long answers cost more (more forward passes)
  • Stopping early saves money (max_tokens)
  • A wrong token early can derail an entire SQL query

KV Cache (Why First Token Is Slowest)

During generation, key/value tensors from previous tokens are cached so they are not recomputed. First token pays full context cost; subsequent tokens are cheaper per step.

Batching and Throughput

APIs batch many user requests on GPUs. Your latency depends on queue depth, model size, output length, and provider load — not just your prompt quality.

sequenceDiagram participant Dev as Developer participant API as LLM API participant GPU as Model on GPU Dev->>API: Prompt (system + user + context) API->>GPU: Tokenize + prefill context loop Until stop token or max_tokens GPU->>GPU: Predict next token distribution GPU->>GPU: Sample / pick token GPU->>API: Stream token API->>Dev: SSE chunk end

Part 7 — Context Window and Position

The context window is the maximum tokens the model can process in one request (input + output combined on most APIs).

Context size (2026 typical)Good forRisk
8K–32KSingle-file edits, short RAGLarge repos need chunking
128K–200KLong docs, big schemas"Lost in the middle" — see Ch 2
1M+ (select models)Whole codebases, legal docsCost, latency, attention dilution

Positional encoding tells the model where each token sits in the sequence. Extended-context models use techniques (RoPE scaling, etc.) to generalize beyond original training length — with varying quality at the extremes.

Best practice

Put rules first, retrieved facts in labeled blocks, user task last. Do not dump 100 pages and hope — rank and compress (Ch 6).

⚠️ Common mistake

Using high temperature for production SQL or JSON because "the model seems more creative." Creativity increases wrong outputs. Use structure + low temperature + schema validation instead.

Part 10 — Chat Format: Roles and Messages

Chat models are trained on multi-turn message lists, not a single blob of text.

RolePurposeExample content
systemStable rules, persona, constraints"You are a SQL assistant. Read-only. PostgreSQL."
developer (OpenAI)App-level instructions (higher priority in some APIs)Tool policies, output schema
userEnd-user task or question"Show unpaid invoices by campus"
assistantPrior model turns in conversationHistory for follow-up questions
toolResults from function callsJSON from run_sql or API

Multi-turn memory: The API is stateless — you resend full message history each call (or use provider-side threads). Long chats eat context and dilute early instructions. Summarize or trim old turns in production.

Part 13 — Structured and Constrained Output

Free-form text is hardest to parse reliably. Production systems constrain the output space.

  • JSON mode / schema: API enforces valid JSON matching a schema
  • Grammar / regex sampling: Force valid SQL subset or enum values
  • Tool calling: Model picks from defined functions with typed args
  • Post-validation: Parse → validate → retry with error message
# Retry loop pattern (simplified)
for attempt in range(3):
    response = llm.generate(prompt, temperature=0.1)
    sql = extract_sql(response)
    err = validate_sql(sql, schema, acl_rules)
    if err is None:
        break
    prompt += f"\nPrevious SQL failed: {err}. Fix only the error."
# Never trust first token stream blindly

Part 20 — Daily Habits for Better LLM Results

  • Start with the smallest model that might work — upgrade only when evals fail
  • Put stable instructions in system message; put changing facts in retrieved context
  • Ask for assumptions explicitly: "List assumptions before SQL"
  • Never merge AI code you cannot explain in one sentence
  • Use @file / RAG instead of describing code from memory
  • Set temperature 0–0.3 for anything that runs in production
  • Log prompts, context version, model ID, and parameters for debugging
  • Re-run critical tasks with a second model or human review
  • Treat the model as a fast intern — skilled supervision required (Ch 23)

Key Takeaway

LLMs predict the next token from patterns in training plus everything in your context window. Best output comes from right model, right parameters, right context, and verification — not from treating the model as an infallible expert. Master the machine; then master the system around it.
